# Per-Product GitHub Repository Design

**Date:** 2026-06-10
**Status:** Draft

---

## Overview

Today the platform pushes every data contract YAML as a file into a single central GitHub repository (`GITHUB_REPO`, default `GuhBrando/dmplt_data_contract`), organized under `contracts/<domain-slug>/<contract-title>.yaml`. The domain only manifests as a folder. Data Product entities in the `data_products` table have no GitHub footprint at all.

This change introduces a dedicated GitHub repository for **each Data Product entity** — to hold its implementation code (pipeline, tests, infrastructure). The central contracts repo is unchanged: contract YAMLs continue to flow there, organized by domain folder, exactly as today.

The new flow is triggered automatically on product creation, and lazily on first read/update of pre-existing products.

---

## Goals

- Every data product has its own private GitHub repo with a standard folder scaffold.
- Repo lifecycle is bound to the product's lifecycle: creation provisions, deletion archives.
- The platform stays usable when GitHub is unreachable (warn-only failures).
- Existing data products are backfilled lazily, without a one-shot migration.

## Non-goals

- Changing where contract YAMLs live (central repo stays as-is).
- Pushing actual pipeline code into the new repos.
- CI workflows or contract-conformance checks in the new repos.
- A separate org/account for product repos (single-account model preserved).
- Per-product permissions/teams setup on GitHub.

---

## Data Model

### `data_products` table

Add one nullable column:

```sql
ALTER TABLE data_products ADD COLUMN repo_url TEXT;
```

Nullable because:
- Existing rows have no repo yet (filled lazily on first access).
- Warn-only creation can fail and leave it null.

### `DataProduct` entity

`backend/domain/entities/data_product.py` gains:

```python
repo_url: str | None = None
```

### `IDataProductRepository`

Adds:

```python
async def update_repo_url(self, product_id: uuid.UUID, repo_url: str) -> None
```

The Postgres implementation issues a single `UPDATE data_products SET repo_url=$1 WHERE id=$2`.

## Naming

Pattern: `dp-<domain-slug>-<product-slug>`, where slugs use the existing `_slugify` helper in `backend/infra/github_client.py` (lowercase, non-alphanumeric collapsed to single dash, edges stripped).

- Domain name is resolved through `data_products → data_contracts → domains` (same join the platform already uses).
- Product slug is derived from `data_product.name`.

**Examples:**

| Domain | Product | Repo name |
|---|---|---|
| `Marketing` | `Attribution Model` | `dp-marketing-attribution-model` |
| `Comunicação Interna` | `Engajamento Comunicados` | `dp-comunicacao-interna-engajamento-comunicados` |
| `Risco de Crédito` | `Default Prediction` | `dp-risco-de-credito-default-prediction` |

The boundary between domain and product is visually ambiguous, but the platform never parses the name back into `(domain, product)` — `repo_url` is stored in the DB. The name is a human label.

Repos are created under the same GitHub account as `GITHUB_REPO` (owner parsed by splitting on `/`). No new config is introduced.

## Visibility & Initial Content

**Visibility:** Private. Matches the README's data mesh architecture — contracts are the public interface, code is private implementation.

**Initial content** (committed as the first commits, since repos are created with `auto_init=false`):

```
README.md
pipeline/.gitkeep
tests/.gitkeep
infrastructure/.gitkeep
```

`README.md` is generated by a helper and contains:

- Product name and description
- Link to the central contract YAML for this product
- Link to the platform's data product detail page
- One-paragraph note explaining the folder layout

The README template lives next to the use case, not inside `GitHubClient`, so the client stays a thin transport layer.

## `GitHubClient` Extensions

`backend/infra/github_client.py` gains three methods:

```python
async def create_product_repo(self, name: str, description: str) -> dict:
    """Creates a private repo under the same owner as GITHUB_REPO.

    Owner type is detected via GET /users/{owner} → 'type': 'User' | 'Organization'.
    Result is cached on the client instance for the process lifetime.
    - Organization → POST /orgs/{owner}/repos
    - User        → POST /user/repos (requires the token to belong to <owner>)
    Body: {private: true, auto_init: false, name, description}.
    Returns {'html_url': ..., 'full_name': '<owner>/<name>'}.
    On 422 'name already exists': GET /repos/{owner}/{name} and return its data
    (adopt path)."""

async def push_scaffold(self, repo_full_name: str, files: dict[str, str]) -> None:
    """For each (path, content) in files, PUT /repos/{full_name}/contents/{path}.
    Reuses the existing push pattern and _raise_for_status."""

async def archive_repo(self, repo_full_name: str) -> None:
    """PATCH /repos/{full_name} with {'archived': true}. Idempotent."""
```

The existing `_repo`, `_headers`, `contract_path`, `push`, `delete`, and `list_all_yaml_contents` are untouched.

## Router Orchestration

`backend/interface/routers/data_product.py` gains a private helper that mirrors `_push_to_github` in the contract router:

```python
async def _ensure_product_repo(
    github: GitHubClient | None,
    product: DataProduct,
    repo: IDataProductRepository,
    domain_name: str,
) -> None:
    if github is None or product.repo_url:
        return
    try:
        name = f"dp-{_slugify(domain_name)}-{_slugify(product.name)}"
        created = await github.create_product_repo(name, product.description)
        await github.push_scaffold(
            created["full_name"], _build_scaffold(product, domain_name)
        )
        await repo.update_repo_url(product.id, created["html_url"])
        product.repo_url = created["html_url"]
    except Exception as exc:
        logger.warning(
            "GitHub repo provision failed for product %s: %s",
            product.id, exc, exc_info=True,
        )
```

`_build_scaffold(product, domain_name)` returns `{path: content}` for `README.md` and the three `.gitkeep` files.

### Per-endpoint wiring

| Endpoint | New behavior |
|---|---|
| `POST /data-products` | After the use case, call `_ensure_product_repo`. Response includes `repo_url` (may be null on failure). |
| `GET /data-products/{id}` | If `repo_url` is null, call `_ensure_product_repo` before responding (lazy backfill). |
| `GET /data-products` (list) | **No** auto-create — avoids N GitHub calls per page. Backfill happens on detail or update. |
| `PUT /data-products/{id}` | If `repo_url` is null after update, call `_ensure_product_repo`. |
| `DELETE /data-products/{id}` | After the use case, if `repo_url` was set, call `github.archive_repo(<full_name parsed from repo_url>)`. Warn-only. |

Use cases (`CreateDataProductUseCase`, etc.) stay pure DB logic. GitHub side-effects live at the router boundary, mirroring how `_push_to_github` works for contracts today.

---

## Failure Modes

All GitHub interactions are **warn-only**:

- Exceptions are caught in `_ensure_product_repo` and `archive_repo` callers, logged with `logger.warning(..., exc_info=True)`.
- The platform's HTTP response still succeeds; `repo_url` is null on failure.
- The next read/update of the product will retry creation via the lazy-backfill path.

**Specific cases:**

| Failure | Handling |
|---|---|
| `GITHUB_TOKEN` missing | `get_github_client` returns `None`; helper returns immediately. |
| GitHub 401/403 (bad token) | Caught, logged, `repo_url` stays null. |
| GitHub 422 "name already exists" | `create_product_repo` fetches the existing repo and returns it (adopt). |
| GitHub 5xx / network | Caught, logged, retried on next access. |
| Scaffold push partial failure | First failed PUT raises; the repo exists but is partial. `repo_url` is **not** stored, so the next access will see "already exists" → adopt → retry the scaffold. Scaffold pushes are idempotent (`PUT contents` with existing SHA). |

## Configuration

No new environment variables. Reuses:

- `GITHUB_TOKEN` — same token used for the central contracts repo. Token scope must include `repo` (full control of private repos) so it can create and archive repos under the account. If the owner is an organization, the token additionally needs the org's repo-creation permission.
- `GITHUB_REPO` — owner is parsed by splitting on `/` and taking the first segment. Only the owner is used here; the repo path is irrelevant for product repo creation. Owner type (user vs organization) is detected at runtime and dispatched to the appropriate GitHub endpoint.

Single Atlas migration: `database/migrations/YYYYMMDDHHMMSS_data_products_repo_url.sql`:

```sql

```

No backfill SQL — existing rows are filled lazily on first detail GET or PUT.

## Testing

**Unit — `GitHubClient`:**
- `create_product_repo` success against a user owner (POSTs to `/user/repos`).
- `create_product_repo` success against an org owner (POSTs to `/orgs/<owner>/repos`).
- Owner-type detection is cached: a second call does not re-fetch `/users/<owner>`.
- `create_product_repo` 422 "name already exists" → fetches existing repo via GET → returns it.
- `create_product_repo` other failures → raises `RuntimeError`.
- `push_scaffold` issues one PUT per file with base64-encoded content.
- `archive_repo` PATCHes `archived: true`.
- Mock `httpx.AsyncClient` the same way the existing tests do.

**Unit — router:**
- `_ensure_product_repo` returns early when `github is None`.
- `_ensure_product_repo` returns early when `product.repo_url` is already set.
- Success path writes `repo_url` via `repo.update_repo_url` and mutates `product.repo_url`.
- Exception from `create_product_repo` is swallowed and logged; `repo_url` stays null.
- `POST /data-products` triggers `_ensure_product_repo` once.
- `GET /data-products/{id}` on a product with null `repo_url` triggers exactly one create; second GET does not.
- `GET /data-products` (list) does **not** trigger creates.
- `PUT /data-products/{id}` on a null-`repo_url` product triggers one create.
- `DELETE /data-products/{id}` with `repo_url` calls `archive_repo`; with null does not.

**Integration tests are out of scope** — the codebase relies on unit tests with mocked `httpx`, matching the existing GitHub test pattern.

## Decisions and Trade-offs

**Why router-level orchestration?**
Matches the existing pattern (`_push_to_github` in the contract router). Use cases stay pure DB logic and easy to unit-test without GitHub mocks. The trade-off — slightly weaker transactional semantics between DB write and GitHub call — is acceptable because failures are warn-only by design.

**Why lazy backfill instead of a sync endpoint?**
The user explicitly chose auto-create-on-next-access. It avoids a separate admin workflow and a long-running batch. The trade-off — mixing read and write side-effects — is contained to product reads only and is bounded (max one create per product over its lifetime).

**Why no list-level backfill?**
A list endpoint with N products would trigger N GitHub `POST /user/repos` calls per page render. That risks rate-limiting and produces unbounded latency. Backfill is intentionally limited to detail/update paths.

**Why adopt-on-422 instead of suffixing names?**
Silent renames (`-2`, `-3`) hide real collisions. Adopting an existing repo of the same name is acceptable because the same token controls both; if the repo is unrelated, the warning is preferable to silent fragmentation.

**Why archive (not delete) on product deletion?**
Code history is preserved, irreversible mistakes are avoided, and GitHub's archive flag is reversible. Deleting a repo to mirror a DB row is a sharp trade-off the user explicitly rejected.

**Why warn-only failures?**
Matches the existing contract-push pattern. Keeps the platform usable during GitHub outages. The lazy-backfill path doubles as a retry mechanism for free.

**Why no new env vars?**
The same token and owner already work. Introducing `GITHUB_PRODUCTS_ORG` or `GITHUB_PRODUCT_REPO_VISIBILITY` was offered and not selected. YAGNI.
